# Product and Technical Sanity Check

## Inputs

- Accept an optional scope path as `$ARGUMENTS`.
- Review the whole repository when no scope is provided.

## Workflow

1. **Reconstruct the app's purpose.** Identify target users and critical workflows.
2. **Evaluate fitness for purpose.** Check architecture, AI usage, and simplicity vs. over-engineering.
3. **Highlight strengths.** Call out sound product instincts and pragmatic choices.
4. **Identify gaps.** Distinguish prototype shortcuts from production blockers.
5. **Describe the better version.** Propose a lean architecture and optimized flow.
6. **Deliver review.** Structure: Purpose, Strengths, Changes, Better Version, Next Steps.

## Rules

- Be direct and fair; focus on product and architecture, not just code style.
- Distinguish prototype compromises from production risks.
- If context is missing, state assumptions and continue.
